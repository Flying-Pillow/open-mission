## Plan: Mission Daemon Worktree Architecture

Create a new architecture specification for a control-repo daemon that manages many missions concurrently, where each executable mission runs in its own Git worktree under `.mission/worktrees/`. The current codebase is a useful reference for transport, mission products, and surface UX, but the spec should treat the existing single-bound daemon, `.mission/missions/active`, branch-default mission selection, and repo-root agent cwd behavior as architecture to replace rather than preserve.

**Immutable Invariants**
1. Language & Tech-Stack Agnostic. Applies to Phase 3. Worktree materialization is strictly a Git operation. The daemon must not include logic for copying `.env` files, running `npm install`, `pnpm install`, `bun install`, or managing package dependencies. Environment hydration belongs to the human developer or to an explicit mission task, never to the orchestrator.
2. Editor Agnostic. Applies to Phases 4 and 6. VS Code is a thin client only. The core daemon and the CLI cockpit must be fully capable of running the entire mission lifecycle without VS Code. No `vscode` API types, extension host dependencies, or editor-specific runtime assumptions may leak into `@mission-control/core` or daemon packages.
3. Agent Agnostic. Applies to Phase 7. Agent session routing must not hardcode GitHub Copilot specifics. The daemon must treat agents as generic compute engines managed through standard runtime interfaces, MCP or ACP style protocols, or PTY or stdio transports so providers such as Copilot, Claude Code, and Aider remain interchangeable behind the same orchestration boundary.

**Steps**
1. Phase 1: write the first full specification draft in `/tmp`, then prepare it to land at `/home/ronald/mission/packages/core/docs/plan - implement multi mission daemon.md` after review. The draft must define terminology and invariants first: control repo root, mission worktree root, mission-local state root, mission id, pending mission, active mission, setup mode, control mode, and worktree mode. This phase blocks all later design work.
2. Phase 2: define the canonical data model and file layout. Keep global settings and the mission registry in the control repo under `.mission/`; place Git worktrees under `.mission/worktrees/<folder-slug>/`; place mission-local control state and artifacts inside the mission worktree under its own `.mission/` directory so product files and task state are worktree-scoped rather than control-repo-scoped. Because `.mission/settings.json` is tracked repo content, each worktree checkout will also contain a copy of that file; the spec must define that this copy is incidental checkout state, while the authoritative settings file for daemon and surface behavior is always the one resolved from the control repo root. The worktree-local `.mission/` directory must not contain daemon runtime state. This phase also needs to define an immutable internal `missionId`, an explicit `mission.tag` metadata field, a stable folder slug that remains unchanged after activation even if issue metadata is added later, and a provider-neutral runtime settings schema that does not auto-detect or privilege a single agent implementation during repository initialization. Depends on 1.
3. Phase 3: define the mission lifecycle in the spec. Cover pending brief creation, mission-tag capture, GitHub issue intake, delayed issue creation for brief-first missions, activation from the current branch into a new mission branch, existing PR/branch adoption, worktree materialization, and completion/cleanup. Activation should create a dedicated mission branch from the current branch unless the operator explicitly adopts an existing PR/branch. If issue creation is unavailable at activation time, Mission may still activate using a temporary `draft` placeholder slug; later issue assignment updates metadata only and must not rename the branch or worktree folder. Worktree materialization must remain strictly a Git operation and explicitly excludes copying `.env` files, installing dependencies, or performing any environment hydration. Depends on 1-2.
4. Phase 4: redesign the daemon contract around a control plane plus mission plane. Replace the single `boundMission` model with a mission registry and mission-context map keyed by `missionId`; remove branch-only mission resolution; split RPCs into control-plane operations that can run without a mission id and mission-plane operations that require one; require all mission execution, session launch, task updates, and status subscriptions to resolve through the mission’s worktree path; and define explicit event/subscription semantics for a multi-mission daemon so clients do not rely on an ambient active mission. This phase must also keep the daemon contract editor-neutral and CLI-first so no VS Code API types, extension host dependencies, or editor-specific runtime assumptions leak into core or daemon packages. Depends on 1-3.
5. Phase 5: define daemon runtime and discovery rules after removing `.mission/daemon/` from the repo. The spec should replace repo-local daemon manifests and repo-local session files with runtime state that lives outside the repository, while keeping deterministic daemon discovery by control-repo identity. This phase must also define how a surface inside a mission worktree resolves the control repo root without confusing the worktree-local checked-out `.mission/settings.json` copy for the authoritative control-repo settings file, and how stop/shutdown works without a repo-local manifest. Depends on 4.
6. Phase 6: define surface behavior for CLI cockpit and VS Code. Replace `repoMode` terminology with workspace/setup terminology aligned to the requested states: not initialized, initialized but not in a mission worktree, initialized and in a mission worktree. The CLI cockpit is a first-class surface and must remain capable of driving the complete lifecycle without VS Code. The cockpit should render `setup` when `.mission/` or `settings.json` is missing or invalid. VS Code should switch the active workspace root to the mission worktree on activation, and the spec must cover the extension restart/open-folder lifecycle plus multi-root behavior when some folders are mission worktrees and others are not. The extension remains a thin client over the same control-plane and mission-plane contract exposed to the CLI. Both CLI and extension must propagate `missionId` for all mission-plane requests. Depends on 2-5. CLI and extension details can be refined in parallel after the protocol and lifecycle are fixed.
7. Phase 7: define routing for agent sessions and automation. All agent sessions, task launches, autopilot turns, console streams, and notifications must be mission-scoped and execute with the mission worktree as cwd. Session ids alone may identify a running session after the daemon resolves ownership internally, but the public contract should remain mission-scoped for mission-plane surface calls. This phase must keep the agent boundary provider-neutral: orchestration uses generic runtime interfaces and transport abstractions, while provider-specific runtimes such as Copilot, Claude Code, or Aider stay behind adapter boundaries rather than hardcoded daemon logic or core settings defaults. Depends on 4-6.
8. Phase 8: define migration and removal boundaries. The spec should explicitly remove `.mission/missions/active`, `.mission/daemon/`, single-mission daemon binding, branch-default selection without mission identity, and any UI or protocol language that still models this as “repo mode”. This phase should also call for `.mission/worktrees/` to be ignored and excluded from root-level workspace scanning to avoid nested-worktree recursion. Depends on 2-7.
9. Phase 9: define the verification matrix for the future implementation. Cover unit tests, integration tests, and manual scenarios for pending missions, delayed issue creation, existing PR/branch adoption, multiple concurrently loaded missions, invalid settings causing setup mode, nested worktree path resolution, daemon shutdown without `.mission/daemon/`, and agent cwd correctness for implementation and verification work. Depends on 3-8.

**Verification**
1. Path-resolution tests: prove that the control repo, a nested `.mission/worktrees/<slug>` mission worktree, and a plain non-initialized repo each resolve to the correct workspace/setup state, control root, and mission worktree root.
2. Contract tests: prove that mission-plane RPCs reject missing `missionId`, while control-plane RPCs for setup, listing, scheduling, and activation remain available without one.
3. Lifecycle tests: cover brief-first pending missions, delayed GitHub issue creation, activation with a temporary `draft` slug, late issue assignment without path or branch rename, issue-first activation, and existing PR/branch adoption without identity drift.
4. Multi-mission daemon tests: run at least two missions concurrently through one daemon and verify isolated status streams, subscriptions, agent sessions, and task mutations.
5. Worktree execution tests: prove that every agent turn, autopilot launch, and verification task runs with the mission worktree as cwd rather than the control repo root.
6. Settings tests: verify that invalid control-repo `.mission/settings.json` content triggers setup mode, that surfaces always resolve the authoritative settings file from the control repo root even when a mission worktree contains its own checked-out `.mission/settings.json` copy, and that worktree-local `.mission/` directories never introduce daemon runtime state.
7. Daemon runtime tests: verify startup, reconnect, and stop flows with no `.mission/daemon/` directory present in the repository.
8. Manual UX checks: initialize from an empty repo, activate a mission, confirm the cockpit enters setup/control/worktree states correctly, confirm VS Code replaces the workspace root with the activated worktree, and verify extension recovery after the workspace switch.
9. Multi-root UX checks: verify behavior when one folder is a mission worktree and another is not, so workspace selection does not drift to the wrong control or mission context.
10. Boundary tests: prove that core and daemon packages import no `vscode` APIs and can run the full lifecycle through CLI and daemon contracts alone.
11. Orchestrator non-hydration tests: prove that activation and worktree materialization never copy `.env` files, install dependencies, or run package-manager commands.
12. Runtime neutrality tests: run the daemon against at least one non-Copilot test runtime through `MissionAgentRuntime` and verify provider-specific logic stays behind adapter boundaries.

**Decisions**
- The working target path for this plan document is `/home/ronald/mission/packages/core/docs/plan - implement multi mission daemon.md`.
- Mission worktrees live under `.mission/worktrees/`.
- Global settings stay in `.mission/settings.json` in the control repo.
- State is split: control-repo settings and registry stay global, mission control state and artifacts move into the mission worktree.
- Worktree-local `.mission/` directories may contain the tracked `.mission/settings.json` checkout because Git materializes tracked repo content in every worktree; however, daemon and surface behavior must always treat the control-repo-root settings file as authoritative.
- Worktree-local `.mission/` directories must not contain daemon runtime state.
- A mission may exist in a pending state before issue number, branch, or worktree creation.
- Brief-first missions may defer issue creation until activation.
- `mission.tag` is explicit mission metadata captured at scheduling time; GitHub labels may inform it, but folder naming does not copy arbitrary labels directly.
- Activation creates a dedicated mission branch from the current branch unless the operator adopts an existing PR/branch.
- Existing PRs or branches can be adopted as missions and should reuse linked issue or PR identity when available.
- If issue creation is unavailable at activation time, Mission may still activate using a temporary `draft` placeholder segment in the folder and branch name.
- Later issue assignment updates mission metadata only; it must not rename the worktree folder or branch.
- If an adopted branch is already checked out in the control repo’s primary worktree, the operator must switch that primary worktree away before creating the dedicated mission worktree.
- VS Code should replace the current workspace root with the mission worktree on activation.
- `.mission/daemon/` is removed entirely from repo-local state.
- Daemon runtime state and live sessions are ephemeral runtime concerns outside the repository.
- The public contract should stop using `repoMode`; setup/control/worktree terminology should replace it.
- The architecture should not preserve compatibility shims for the current single-bound daemon model.
